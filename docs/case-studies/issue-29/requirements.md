# Requirements extracted from issue #29

The issue body packs several requirements into a few sentences. Each is split out below with an explicit acceptance criterion.

## R1 — Every demo is listed in the README

> *"Make sure all our demos are listed in README"*

**Acceptance:** `README.md` contains a *Demos* section that lists every demo HTML file in the repository, with a one-line description and a link that works on the published Pages site.

**Inventory of demos in the current `main`:**

| Demo                              | File                              |
|-----------------------------------|-----------------------------------|
| Entity Viewer                     | `entities.html`                   |
| Property Viewer                   | `properties.html`                 |
| Text → Q/P Transformer            | `transformation/index.html`       |
| N-gram Transformer Test           | `transformation/test-ngram.html`  |
| Search & Disambiguation Demo      | `search-demo.html`                |
| Caching Demo                      | `cache-demo.html`                 |
| Browser Cache Test                | `browser-cache-test.html`         |
| Test Runner (browser)             | `run-tests.html`                  |

## R2 — Demos are published in GitHub Actions

> *"… they are published in GitHub Actions …"*

**Acceptance:** A successful GitHub Actions run publishes every file in the inventory above to `https://link-assistant.github.io/human-language/<path>` and the URLs return HTTP 200.

**Current state:** The implicit `pages-build-deployment` workflow already publishes every file in the repository root via Jekyll. All eight demos in the inventory return HTTP 200. The remaining gap is *triggering the republish* once we change the README/index — that happens automatically on `main` push.

## R3 — The main page links directly to every demo

> *"… we have direct links to all demos from https://link-assistant.github.io/human-language main page"*

**Acceptance:** The HTML served at `https://link-assistant.github.io/human-language/` contains a clearly-labelled link (anchor with visible text) to each demo URL.

**Current state:** No `index.html` exists. Jekyll renders `README.md`. The README mentions some demos by filename ("Open `entities.html`") but does **not** provide working hyperlinks because GitHub Pages serves files directly and Jekyll does not auto-link bare filenames in markdown text.

## R4 — Clean up rename leftovers

> *"Also double check for any left overs after rename from deep-assistant to link-assistant …"*

**Acceptance:** No file in the repository links to `https://github.com/deep-assistant/...` or `https://deep-assistant.github.io/...` unless that link is intentional historical context. The published Jekyll page also no longer renders `deep-assistant` URLs in canonical / og:url tags.

**Current state:** `README.md` contains 9 `deep-assistant` URLs; the Jekyll output also produces a `<link rel="canonical" href="https://deep-assistant.github.io/human-language/" />` tag that is generated by Jekyll's `jekyll-seo-tag` plugin.

## R5 — Fix bugs that prevent correct republish

> *"… and fix some bugs to correctly trigger republish."*

**Acceptance:** Pushing a change to `main` produces a fresh, *correct* deployment — i.e. the served page reflects the change, and the served HTML contains the right canonical URL (`link-assistant.github.io`, not `deep-assistant.github.io`).

**Root causes identified** (see `root-causes.md`): missing `_config.yml` (Jekyll has no project URL configured, so `jekyll-seo-tag` infers an outdated one), and the README's hard-coded `deep-assistant` links survive the publish unchanged.

## R6 (process) — Compile this case study

> *"We need to download all logs and data related about the issue to this repository, make sure we compile that data to ./docs/case-studies/issue-{id} folder, and use it to do deep case study analysis …"*

**Acceptance:** This folder (`docs/case-studies/issue-29/`) exists and contains the raw data dumps and the analysis documents listed in `README.md` here.

## R7 (process) — Add tracing if root cause cannot be found

> *"If there is not enough data to find actual root cause, add debug output and verbose mode if not present …"*

**Status:** Not required for this issue — every problem has a clear root cause already (see `root-causes.md`), and the relevant log streams (`pages-builds.json`, `workflow-runs.json`) are already exhaustive.

## R8 (process) — File upstream issues if applicable

> *"If issue related to any other repository/project, where we can report issues on GitHub, please do so."*

**Status:** No upstream bug. The behaviour we're seeing in `jekyll-seo-tag` is the documented behaviour: it derives `site.url` from a heuristic when `_config.yml` doesn't supply one, and on a renamed repo that heuristic is wrong. The fix is configuration on our side.
